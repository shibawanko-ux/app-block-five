# オンライン対戦 要件定義 v0.1

> 作成日：2026-06-21　担当：たくやさん + つむぎ
> ステータス：**v0.2 確定**

---

## 1. 目的・背景

現在の「2人対戦」モードは同一デバイス限定。  
離れた場所にいる相手とブラウザだけで対戦できるようにする。

---

## 2. オンライン対戦のゲームフロー

```
P1 がゲームを作成（ルーム生成）
  └─ 4桁ルームコードを取得
      └─ P2 にコードを共有（コピーボタン or URL）
          └─ P2 がコードを入力 → 対戦開始

P1 がブロックを落とす（クリック）
  └─ 自分の画面でリアルタイム物理演算
      └─ ブロックが静止したら「settle イベント」を送信
          └─ P2 の画面にブロックが反映される
              └─ P2 のターンへ

勝敗判定 → WIN/DRAW → どちらからでも「もう一回」要求可
```

---

## 3. ルーム管理仕様

| 項目 | 仕様 |
|------|------|
| ルームコード | 数字4桁（例：`7392`）← たくやさん確認済み |
| URL 共有 | なし。コード入力のみ ← たくやさん確認済み |
| 人数 | 2人固定 |
| 3人目以降 | 拒否（観戦モードはスコープ外） |
| ルーム有効期限 | 最終操作から 30分（BaaS の TTL 設定） |
| ルームの再利用 | ゲーム終了後に同一ルームで再戦可 |

---

## 4. 同期設計（何を送るか）

### 方針：「settle イベントのみ同期」（MVP 採用）

物理演算は各クライアントが独自に実行する。  
送信するのは「ブロックが確定した位置（列・行）」だけ。

| イベント | 送信者 | データ | 受信側の動作 |
|---------|--------|--------|------------|
| `drop` | 操作プレイヤー | `{ x: number }` | 相手画面でも物理落下アニメーション再生 ✅ MVP に含む |
| `settle` | 操作プレイヤー | `{ col, row, player }` | 相手のボードを権威的に更新（ズレ補正） |
| `restart` | どちらでも | `{ accepted: boolean }` | 両者合意でリスタート |
| `disconnect` | サーバー自動検出 | — | 「相手が切断しました」表示 |

### drop + settle 両方同期の理由（たくやさん確認済み）

- **drop を同期** → 相手がブロックを落とす瞬間のアニメーションを両画面で見られる
- **settle で補正** → 物理演算の浮動小数点差があっても盤面は settle で一致する
- drop → settle の流れで「体験の豊かさ」と「データ整合性」を両立する

## 5. 技術スタック選定

### 推奨：Firebase Realtime Database

| 評価項目 | Firebase | Supabase | PeerJS (WebRTC) |
|---------|----------|----------|-----------------|
| バックエンドサーバー不要 | ✅ | ✅ | ✅（シグナリングは外部） |
| 無料枠 | ✅ 十分 | ✅ 十分 | ✅（PeerJS の公開サーバー） |
| セットアップ速度 | ⭐⭐⭐ 速い | ⭐⭐ 普通 | ⭐ やや面倒 |
| 接続安定性 | ⭐⭐⭐ 高い | ⭐⭐⭐ 高い | ⭐⭐ NAT 越えに難 |
| React SDK の充実度 | ⭐⭐⭐ | ⭐⭐⭐ | ⭐ |
| Vercel との相性 | ✅ | ✅ | ✅ |

**推奨理由：** Firebase の Realtime Database は WebSocket ベースで低遅延、  
Vite + React から `firebase` パッケージ 1つで使える。  
Vercel（静的ホスティング）に追加のバックエンドなしで動く。

## 8. 非機能要件

| 項目 | 目標値 |
|------|--------|
| 操作から相手画面への反映 | 500ms 以内（国内）|
| ルームコードの衝突率 | 4桁 = 1,679,616 通り → 実用上無視可 |
| Firebase 利用料（個人） | 無料枠（Spark Plan）で十分：同時接続 100、1GB/月 |
| ブラウザ対応 | Chrome / Safari / Edge（最新版） |

---

## 9. 実装上の注意点

- P1 / P2 の割り当ては「先に入ったほうが P1」（Firebase の接続順で決定）
- 自分のターン以外はクリックを無効化（現在の CPU 対戦ロジックを流用可）
- 物理演算のズレ補正：settle 受信時に `boardRef` を強制上書きする
- ルームコード生成：`Math.random()` ではなく Firebase の push ID または crypto.randomUUID() の先頭6文字

## 11. Firebase Realtime Database セキュリティルール設定手順

### 11.1 なぜ設定が必要か

Firebase の新規プロジェクトはデフォルトで **テストモード（30日間限定 read/write 許可）** で作成される。  
期限が切れると全リクエストが拒否されてゲームが動かなくなるため、本番ルールへ移行する必要がある。

### 11.2 設定箇所

Firebase Console → 左メニュー「構築」→「Realtime Database」→ 上部タブ「ルール」

### 11.3 推奨ルール（MVP用）

```json
{
  "rules": {
    "rooms": {
      "$roomCode": {
        ".read": true,
        ".write": true
      }
    }
  }
}
```

**この設定の意味：**

- `rooms/{4桁コード}` 配下のみ read/write を許可
- ルームコードを知っている人だけがアクセスできる（総当たりは実用上困難）
- それ以外のパス（`/users` 等）へのアクセスは拒否

### 11.4 設定手順（ブラウザ操作）

1. [Firebase Console](https://console.firebase.google.com/) を開く
1. プロジェクト `block-five` を選択
1. 左メニュー「構築」→「**Realtime Database**」をクリック
1. 上部タブ「**ルール**」をクリック
1. 既存のルールを以下に書き換える：

```json
{
  "rules": {
    "rooms": {
      "$roomCode": {
        ".read": true,
        ".write": true
      }
    }
  }
}
```

1. 「**公開**」ボタンを押す

### 11.5 将来的なルール強化（v0.2 以降）

現状のルールはルームコード知っていれば誰でも書き込める。  
本格運用する場合は以下を検討：

```json
{
  "rules": {
    "rooms": {
      "$roomCode": {
        ".read": "data.child('hostConnected').val() === true || data.child('guestConnected').val() === true",
        ".write": true,
        "lastEvent": {
          ".validate": "newData.hasChildren(['type', 'player', 'ts'])"
        }
      }
    }
  }
}
```

### 11.6 古いルームデータの自動削除（TTL 設定）

Firebase Realtime Database 単体には TTL 機能がない。  
古いルームは `removeRoom()` 関数（ゲーム終了・キャンセル時）が呼ばれると削除される。  
手動クリーンアップが必要な場合は Firebase Console → Realtime Database → データタブで確認可能。
